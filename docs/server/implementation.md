# Реализация: стек, архитектура и первый этап

[Docs](../README.md) → [Server](README.md) → **implementation**

Технические решения и чек-лист для разработки. Продуктовый контекст — [PRD](../product/PRD.md). Контракт API — [API](api/README.md) (версия v1, префикс `/v1`).

---

## 1. Стек технологий

**Frontend**
- React, TypeScript, Vite, SCSS.
- Роутинг: React Router v7.
- Telegram: `@telegram-apps/sdk-react` (launchParams, тема, MainButton, BackButton при необходимости).
- Стили: SCSS, модульная структура (например, `*-page.scss`, компонентные стили).

**Backend**
- **NestJS** (TypeScript) — единый язык с фронтом, удобная структура и встроенная поддержка JWT, guards, валидации.
- **Prisma** — ORM с отличной типизацией, автогенерацией типов из схемы, удобными миграциями.
- **PostgreSQL** — реляционная БД для пользователей, прогресса, заданий, подписок.
- **RESTful API** — все взаимодействия по REST (JSON, стандартные коды ответов, ресурсная модель). **Версионность API обязательна:** для начала **v1** (префикс `/v1/...`); при домене вида `api.example.com` путь не дублирует «api». Контракт — в [API](api/README.md).

## 3. Роуты приложения (фронт)

- **При открытии в браузере (вне Telegram):** `/` или `/welcome` — только [посадочная страница](../client/screens/landing.md): описание продукта и **переход к боту** (через которого запускается Mini App). Полноценного входа в приложение из браузера на первом этапе **нет**.
- **Основное приложение:** только в **Telegram Mini App** — `/` главная (маскот, настроение); список тем и уроков — через нижнее меню. **В планах:** авторизация в браузере (Telegram Login Widget и т.п.), чтобы тот же SPA работал и после входа с посадочной; см. [API: авторизация, планы](api/auth.md).
- `/topics` или раздел **«Темы»** в нижнем меню — список тем; по клику — уроки выбранной темы.
- `/topics/:topicId/lessons` — уроки внутри темы (или встроено в один экран со списком).
- `/lessons/:lessonId` или `/topics/:topicId/lessons/:lessonId` — тренажёр (упражнения выбранного урока).
- `*` — страница ошибки (404).

## 4. Авторизация (JWT) и защищённые роуты

- **Модель токенов:** **`accessToken`** — JWT (короткий срок); **`refreshToken`** — непрозрачная строка, хранится в БД на сервере, в ответе API поле `refreshToken`. Зачем не «два JWT», ротация `refreshToken`, отзыв сессии — подробно в [API: авторизация, раздел «Модель токенов»](api/auth.md#модель-токенов-access-jwt-и-refresh-opaque).
- **`accessToken`** и **`refreshToken`:** после входа через Telegram бэкенд выдаёт **`accessToken`** (рекомендуемый срок жизни 30 мин) и **`refreshToken`** (сессия до 30 дней, запись в БД). Клиент передаёт **`accessToken`** в заголовке `Authorization: Bearer <accessToken>` при запросах к API. При 401 по защищённому роуту клиент вызывает `POST /v1/auth/refresh` с **`refreshToken`**; при успехе сохраняет новую пару и повторяет запрос. Повторный вход через Telegram — только при истечении или невалидности **`refreshToken`** (см. [PRD, раздел 4.5](../product/PRD.md)).
- **Защищённые эндпоинты:** требуют валидного JWT (**`accessToken`**). Публичные — `POST /v1/auth/login`, `POST /v1/auth/refresh` (см. [API: авторизация](api/auth.md)).
- **Клиент (Mini App):** хранение **`accessToken`** и **`refreshToken`** безопасно (память и т.д. в WebView); при 401 после неудачного обновления по **`refreshToken`** — очистка **`accessToken`** и **`refreshToken`**, экран с переходом к боту (не «вход с посадочной», т.к. браузерной сессии на первом этапе нет).

## 5. Бэкенд и база данных

- **NestJS + PostgreSQL** для хранения данных пользователя: выполненные упражнения, прогресс по урокам и темам, статистика, зернышки, достижения, настройки (в т.ч. автопрослушивание). Единая картина с любого устройства.
- **Интеграции:** Telegram Web App (launchParams, themeParams); REST API для авторизации (JWT по `initData`), прогресса, контента, аудио.

### 5.1 Тренажёр: логика выбора упражнения и хранение прогресса

- **Один запрос — одно упражнение.** Эндпоинт `GET /v1/lessons/:lessonId/exercises/next` возвращает одно упражнение. Сервер выбирает его по правилам (см. [PRD](../product/PRD.md), п. 3.4.1):
  - **Пока не все упражнения урока выполнены пользователем хотя бы раз:** возвращать только упражнения без записи о выполнении для этого пользователя (по БД).
  - **Когда все упражнения урока выполнены:** возвращать любое упражнение урока в случайном порядке, исключая **последние 5 выполненных** в этом уроке (чтобы не повторяться подряд). Для этого хранить порядок или время последних выполненных по паре «пользователь + урок».
- **Проверка ответа** выполняется на бэкенде: эндпоинт `POST /v1/lessons/:lessonId/exercises/:exerciseId/answer` принимает ответ, сравнивает с эталоном (с учётом типа: сбор предложения, выбор варианта и т.д.), записывает факт выполнения в БД и возвращает `correct` и при необходимости подсказку/пояснение.
- **Хранение в БД (рекомендуемая модель):**
  - **Таблица выполнений** (например, `exercise_completions`): `user_id`, `lesson_id`, `exercise_id`, `correct` (boolean), `completed_at` (timestamp). По ней считаются прогресс урока, «первое прохождение» для антифарма зернышек и список «последние 5 по уроку» для исключения при выдаче следующего упражнения.
  - Для правила «последние 5 не выдавать» при свободном режиме: по `user_id` и `lesson_id` выбирать последние 5 записей по `completed_at`, получать их `exercise_id` и исключать из выборки при случайном выборе следующего упражнения.
- Детали контракта запросов/ответов — в [API](api/README.md).

## 6. Рекомендации для улучшения стека

- **Валидация:** на бэке — `class-validator` + `class-transformer` (NestJS); на фронте — Zod или Yup для форм и ответов API.
- **Работа с API на фронте:** React Query (TanStack Query) или SWR — кэш, повторные запросы, инвалидация после мутаций (прогресс, зернышки).
- **ORM/миграции:** Prisma — типизированные модели из `schema.prisma`, миграции через `prisma migrate`.
- **Конфигурация:** переменные окружения (`.env`) на фронте и бэке; на бэке — `@nestjs/config`; не коммитить секреты (JWT secret, ключи Telegram, БД).
- **Тесты:** Jest (NestJS и Vite); unit для сервисов и валидации; e2e для критичных сценариев (логин, отправка ответа на задание).
- **Деплой:** Docker для NestJS и при необходимости фронта; PostgreSQL в контейнере или управляемый сервис.

## 8. Первый этап: базовый бэк + авторизация в Mini App

Цель: базовый бэкенд (авторизация по **initData** из Telegram, JWT, защищённый роут `GET /v1/topics`) и клиент Mini App с защищёнными запросами. **Браузерная авторизация** (вход с посадочной без Mini App) **не входит** в этот этап — см. [api/auth.md](api/auth.md), раздел «Планы».

**Чек-лист для старта:**

| # | Что сделать | Заметки |
|---|-------------|---------|
| 1 | **Контракт API** | В [api/README.md](api/README.md) по разделам: авторизация — [api/auth.md](api/auth.md) (`POST /v1/auth/login`, `POST /v1/auth/refresh`), [topics](api/topics.md), тренажёр. При изменении — обновлять соответствующие файлы и Swagger. |
| 2 | **Переменные окружения** | Бэк: `JWT_SECRET`, `TELEGRAM_BOT_TOKEN` (проверка initData), `DATABASE_URL`. Клиент: `VITE_API_URL` (хост, без `/v1`). Реальные значения — в `.env`, не коммитить. Шаблоны — в `frontend/.env.example` и `backend/.env.example`; при добавлении новых env оба файла обновлять (см. PRD, раздел 14). |
| 3 | **Бэкенд: проект NestJS** | Инициализация, PostgreSQL (TypeORM/Prisma), модули: Auth (проверка Telegram initData, выдача JWT), Topics (`GET /v1/topics` — мок или 2–3 темы из БД). JWT Guard на защищённые роуты. |
| 4 | **Бэкенд: документация** | Swagger/OpenAPI для всех роутов; JSDoc/TSDoc для сервисов и методов (п. 7 выше). |
| 5 | **Клиент: хранение токенов (Mini App)** | После успешного `POST /v1/auth/login` сохранять поля `accessToken` и `refreshToken`; в запросах к API — заголовок `Authorization: Bearer <accessToken>`. При 401 — попытка обновления через `POST /v1/auth/refresh`; при успехе — повторить запрос, при неудаче — очистка **`accessToken`** и **`refreshToken`**, затем UI с переходом к боту. |
| 6 | **Клиент: сценарий авторизации** | **Только Mini App:** при старте отправить `initData` на бэк, получить **`accessToken`** и **`refreshToken`**, сохранить. **Браузер:** только посадочная и ссылка на бота — без `login` до отдельной фичи. |
| 7 | **Клиент: защищённые запросы** | Запросы с **`accessToken`** (`GET /v1/topics` и др.). При 401 — сначала `POST /v1/auth/refresh`; при неудаче — очистка **`accessToken`** и **`refreshToken`**, экран с ботом (Mini App). |
| 8 | **Посадочная (браузер)** | Контент + CTA на бота (открытие Mini App); без рабочего приложения вне Telegram на этом этапе. |
| 9 | **Данные для GET /v1/topics** | Формат по [api/topics.md](api/topics.md) (id, name, description, order); пока захардкодить или 2–3 темы в БД. |

Для этого этапа достаточно описания в PRD и контракта в api/README.md. Детали (формат initData, поля сущности Topic) уточнять по ходу.
