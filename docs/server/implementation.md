# Реализация: стек, архитектура и первый этап

[Docs](../README.md) → [Server](README.md) → **implementation**

Технические решения и чек-лист для разработки. Продуктовый контекст — [PRD](../product/PRD.md). Контракт API — [API](api/README.md) (версия v1, префикс `/v1`).

---

## 1. Стек технологий

**Frontend**
- React, TypeScript, Vite, SCSS.
- Роутинг: React Router v7.
- Telegram: `@telegram-apps/sdk-react` (launchParams, тема, MainButton, BackButton при необходимости).
- Стили: SCSS, модульная структура (например, `*-page.scss`, компонентные стили).

**Backend**
- **NestJS** (TypeScript) — единый язык с фронтом, удобная структура и встроенная поддержка JWT, guards, валидации.
- **Prisma** — ORM с отличной типизацией, автогенерацией типов из схемы, удобными миграциями.
- **PostgreSQL** — реляционная БД для пользователей, прогресса, заданий, подписок.
- **RESTful API** — все взаимодействия по REST (JSON, стандартные коды ответов, ресурсная модель). **Версионность API обязательна:** для начала **v1** (префикс `/v1/...`); при домене вида `api.example.com` путь не дублирует «api». Контракт — в [API](api/README.md).

## 3. Роуты приложения (фронт)

- **При открытии в браузере (без авторизации):** `/` или `/welcome` — посадочная страница с кнопкой «Войти через Telegram». После авторизации — переход в основное приложение.
- **Основное приложение (в Telegram или после входа в браузере):** `/` — главная (маскот, настроение); список тем и уроков — через нижнее меню.
- `/themes` или раздел из нижнего меню — список тем; по клику на тему — список уроков темы.
- `/themes/:topicId/lessons` — уроки внутри темы (или встроено в один экран со списком тем).
- `/lessons/:lessonId` или `/themes/:topicId/lessons/:lessonId` — тренажёр (упражнения выбранного урока).
- `*` — страница ошибки (404).

## 4. Авторизация (JWT) и защищённые роуты

- **Access и refresh токены:** после входа через Telegram бэкенд выдаёт access token (рекомендуемый срок жизни 30 мин) и refresh token (30 дней). Клиент передаёт access в заголовке `Authorization: Bearer <accessToken>` при запросах к API. При 401 по защищённому роуту клиент вызывает `POST /v1/auth/refresh` с refresh токеном; при успехе сохраняет новую пару и повторяет запрос. Повторный вход через Telegram — только при истечении или невалидности refresh (см. [PRD, раздел 4.5](../product/PRD.md)).
- **Защищённые эндпоинты:** требуют валидного access JWT. Публичные — `POST /v1/auth/login`, `POST /v1/auth/refresh`; выход — `POST /v1/auth/logout` (см. [API: авторизация](api/auth.md)).
- **Клиент:** хранение access и refresh токенов безопасно (память, sessionStorage или httpOnly cookie в веб-версии); при 401 после неудачного refresh — очистка токенов и редирект на посадочную.

## 5. Бэкенд и база данных

- **NestJS + PostgreSQL** для хранения данных пользователя: выполненные задания, прогресс по урокам и темам, статистика, зернышки, достижения, настройки (в т.ч. автопрослушивание). Единая картина с любого устройства.
- **Интеграции:** Telegram Web App (launchParams, themeParams); REST API для авторизации (JWT), прогресса, контента, аудио; авторизация через Telegram (hash/auth_date) для выдачи JWT.

### 5.1 Тренажёр: логика выбора задания и хранение прогресса

- **Один запрос — одно задание.** Эндпоинт `GET /v1/lessons/:lessonId/tasks/next` возвращает одно задание. Сервер выбирает его по правилам (см. [PRD](../product/PRD.md), п. 3.4.1):
  - **Пока не все задания урока выполнены пользователем хотя бы раз:** возвращать только задания без записи о выполнении для этого пользователя (по БД).
  - **Когда все задания урока выполнены:** возвращать любое задание урока в случайном порядке, исключая **последние 5 выполненных** в этом уроке (чтобы не повторяться подряд). Для этого хранить порядок или время последних выполненных заданий по паре «пользователь + урок».
- **Проверка ответа** выполняется на бэкенде: эндпоинт `POST /v1/lessons/:lessonId/tasks/:taskId/answer` принимает ответ, сравнивает с эталоном (с учётом типа задания: сбор предложения, выбор варианта и т.д.), записывает факт выполнения в БД и возвращает `correct` и при необходимости подсказку/пояснение.
- **Хранение в БД (рекомендуемая модель):**
  - **Таблица выполнений заданий** (например, `task_completions` или `user_task_progress`): `user_id`, `lesson_id`, `task_id`, `correct` (boolean), `completed_at` (timestamp). По ней считаются прогресс урока (сколько заданий решено), «первое прохождение» для антифарма зернышек и список «последние 5 по уроку» для исключения при выдаче следующего задания.
  - Для правила «последние 5 не выдавать» при свободном режиме: по `user_id` и `lesson_id` выбирать последние 5 записей по `completed_at` (или по отдельной таблице сессии), получать их `task_id` и исключать из выборки при случайном выборе следующего задания.
- Детали контракта запросов/ответов — в [API](api/README.md).

## 6. Рекомендации для улучшения стека

- **Валидация:** на бэке — `class-validator` + `class-transformer` (NestJS); на фронте — Zod или Yup для форм и ответов API.
- **Работа с API на фронте:** React Query (TanStack Query) или SWR — кэш, повторные запросы, инвалидация после мутаций (прогресс, зернышки).
- **ORM/миграции:** Prisma — типизированные модели из `schema.prisma`, миграции через `prisma migrate`.
- **Конфигурация:** переменные окружения (`.env`) на фронте и бэке; на бэке — `@nestjs/config`; не коммитить секреты (JWT secret, ключи Telegram, БД).
- **Тесты:** Jest (NestJS и Vite); unit для сервисов и валидации; e2e для критичных сценариев (логин, отправка ответа на задание).
- **Деплой:** Docker для NestJS и при необходимости фронта; PostgreSQL в контейнере или управляемый сервис.

Цель: базовый бэкенд (авторизация через Telegram, JWT, защищённый роут со списком тем) и доработка клиента для работы с авторизацией и защищёнными запросами.
