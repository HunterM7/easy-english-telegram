# API — контракт (v1)

[Docs](../../README.md) → [Server](../README.md) → **API**

**Контракт** — это договорённость между клиентом (фронтом) и бэкендом: какие есть адреса (роуты), какой запрос отправлять (метод, заголовки, тело) и какой ответ ждать (коды, формат полей). Когда контракт зафиксирован в документе (или в OpenAPI), фронт и бэк могут разрабатываться параллельно и меньше ломать друг друга при изменениях. Ниже — общие правила; описание эндпоинтов разбито по файлам в этой папке.

---

## Версионность

Все роуты относятся к версии **v1** и имеют префикс:

**Базовый путь:** `/v1`

Пример: авторизация — `POST /v1/auth/login`, список тем — `GET /v1/themes`.

Если домен уже указывает на API (например, `api.example.com`), путь версии не дублирует «api»: полный URL — `https://api.example.com/v1/...`. На клиенте в `VITE_API_URL` задаётся хост (например, `https://api.example.com`), к путям запросов добавляется префикс `/v1`.

При появлении несовместимых изменений вводится новая версия (например, `/v2`); v1 по согласованию можно поддерживать или отключать.

---

## Общие условия

- Базовый URL хоста: `https://api.example.com/` (или `VITE_API_URL` на клиенте). К путям ниже добавляется префикс `/v1`.
- Все ответы в формате JSON. Кодировка: UTF-8.

## Общие правила

- **Защищённые роуты:** все эндпоинты кроме `POST /v1/auth/login`, `POST /v1/auth/logout` и `POST /v1/auth/refresh` требуют заголовка `Authorization: Bearer <accessToken>`. Подробно — в [auth.md](auth.md).
- **При 401:** клиент вызывает `POST /v1/auth/refresh` с сохранённым refresh токеном; при успехе — сохранить новые токены и повторить запрос; при 401 от refresh — очистить токены и перенаправить на посадочную (вход через Telegram).
- **Валидация:** на бэкенде проверять тип и наличие обязательных полей; при ошибке возвращать 400 с полем `message` (и при необходимости `errors` со списком полей).
