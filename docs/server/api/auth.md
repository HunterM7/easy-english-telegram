# API: Авторизация

[Docs](../../README.md) → [Server](../README.md) → [API](README.md) → **Авторизация**

В приложении Easy English единственный способ входа — **авторизация через Telegram**. Пользователь входит в Mini App из Telegram или на посадочной странице в браузере нажимает «Войти через Telegram». Бэкенд проверяет данные Telegram и выдаёт пару JWT: **access token** и **refresh token**. Дальнейшие запросы к API выполняются с access токеном в заголовке; при его истечении клиент обновляет сессию по refresh токену, не требуя повторного входа.

---

## Схема работы

### Вход (получение токенов)

1. **Mini App (открытие из Telegram):** при старте приложения клиент получает от Telegram Web App строку `initData`, отправляет её на бэкенд в теле `POST /v1/auth/login`. Сервер проверяет подпись (валидность данных от Telegram), создаёт или находит пользователя в БД и возвращает `accessToken` и `refreshToken`.
2. **Браузер (посадочная страница):** пользователь нажимает «Войти через Telegram», проходит авторизацию в Telegram (Login Widget или редирект в Telegram), после чего клиент получает данные пользователя и строку для верификации; отправляет их на `POST /v1/auth/login` и получает ту же пару токенов.

Клиент сохраняет оба токена (в памяти, sessionStorage или безопасном хранилище) и при каждом запросе к защищённым эндпоинтам передаёт access токен в заголовке `Authorization: Bearer <accessToken>`.

### Сроки жизни токенов

| Токен   | Рекомендуемый срок | Назначение |
|--------|---------------------|------------|
| Access  | 30 минут            | Подпись запросов к API; короткий срок снижает риски при утечке. |
| Refresh | 30 дней             | Обновление access без повторного входа; при истечении или отзыве пользователь снова входит через Telegram. |

Конкретные значения задаются на бэкенде (переменные окружения или конфиг).

### Обновление сессии (refresh)

Когда защищённый эндпоинт возвращает **401 Unauthorized** (access токен истёк или невалиден), клиент **не** перенаправляет пользователя на вход сразу. Он:

1. Отправляет запрос **`POST /v1/auth/refresh`** с телом `{ "refreshToken": "..." }` (значение сохранённого refresh токена).
2. Если ответ **200 OK** — сохраняет новые `accessToken` и при необходимости `refreshToken` из ответа, затем **повторяет исходный запрос** к API с новым access токеном.
3. Если ответ **401** (refresh истёк или невалиден) — очищает сохранённые токены и перенаправляет пользователя на посадочную страницу (вход через Telegram).

Таким образом, пользователь повторно входит только когда истекает или отзывается refresh токен, а не при каждом истечении access.

### Публичные и защищённые эндпоинты

- **Публичные** (без access-токена): `POST /v1/auth/login`, `POST /v1/auth/refresh`. Эндпоинт `POST /v1/auth/logout` принимает refresh-токен в теле запроса (см. ниже).
- **Защищённые**: все остальные эндпоинты API требуют заголовка `Authorization: Bearer <accessToken>`.

## Эндпоинты

### POST /v1/auth/login

Вход в приложение. Обмен данных верификации Telegram на пару JWT (access + refresh). Вызывается после того, как пользователь авторизовался через Telegram (в Mini App — при старте, в браузере — после нажатия «Войти через Telegram» и успешной авторизации в Telegram).

**Запрос**

| Поле       | Тип    | Обязательно | Описание |
|------------|--------|-------------|----------|
| `initData` | string | да          | Строка initData от Telegram Web App. Содержит данные пользователя и подпись; бэкенд проверяет подпись с помощью секрета бота (см. документацию Telegram). |

Пример тела:

```json
{
  "initData": "query_id=AAHdF6IQAAAAAN0XohAOq..."
}
```

**Успешный ответ** — `200 OK`

| Поле           | Тип    | Описание |
|----------------|--------|----------|
| `accessToken`  | string | JWT для заголовка `Authorization: Bearer <token>`. Срок жизни — 30 мин (рекомендуется). |
| `refreshToken` | string | JWT для обновления access через `POST /v1/auth/refresh`. Срок жизни — 30 дней (рекомендуется). |
| `expiresIn`    | number | (опционально) Время жизни access токена в секундах. |
| `user`        | object | (опционально) Данные пользователя для отображения на клиенте. |

Пример ответа:

```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "expiresIn": 1800,
  "user": {
    "id": "telegram-user-id",
    "firstName": "Имя",
    "lastName": "Фамилия",
    "username": "username",
    "photoUrl": "https://..."
  }
}
```

**Ошибки**

| Код | Описание |
|-----|----------|
| 400 | Неверный или поддельный initData (проверка подписи не прошла). |
| 401 | Не удалось верифицировать пользователя Telegram. |
| 500 | Внутренняя ошибка сервера. |

Тело ошибки (единый формат API):

```json
{
  "statusCode": 400,
  "message": "Invalid Telegram initData",
  "error": "Bad Request"
}
```

### POST /v1/auth/logout

Выход из приложения. Клиент отправляет текущий refresh-токен; сервер помечает его как недействительный (чёрный список или удаление из хранилища активных сессий), чтобы им нельзя было получить новый access. После вызова клиент очищает сохранённые токены и (в браузерной версии) перенаправляет на посадочную страницу — пользователь может войти через другой аккаунт Telegram.

**Когда вызывать:** при нажатии пользователем кнопки «Выйти» в настройках (в браузерной версии кнопка показывается в Настройках; в Mini App — по желанию, т.к. смена аккаунта там менее типична).

**Запрос**

| Поле           | Тип    | Обязательно | Описание |
|----------------|--------|-------------|----------|
| `refreshToken` | string | да         | Текущий refresh token (тот, который нужно инвалидировать). |

Пример тела:

```json
{
  "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
```

**Успешный ответ** — `200 OK` или `204 No Content`

Тело ответа пустое или `{}`. Даже при невалидном/уже истёкшем refresh сервер может вернуть 200, чтобы клиент в любом случае очистил токены.

**Ошибки**

| Код | Описание |
|-----|----------|
| 400 | Нет refreshToken в теле. |
| 500 | Внутренняя ошибка сервера. |

После вызова logout клиент всегда удаляет access и refresh из хранилища и перенаправляет на посадочную (в браузере).

### POST /v1/auth/refresh

Обновление access токена по действительному refresh токену. Вызывается клиентом при получении 401 от любого защищённого эндпоинта: сначала попытка обновления; при успехе — повтор исходного запроса; при 401 от refresh — очистка токенов и редирект на посадочную (вход через Telegram).

**Запрос**

| Поле           | Тип    | Обязательно | Описание |
|----------------|--------|-------------|----------|
| `refreshToken` | string | да         | Refresh token, полученный при входе (`POST /v1/auth/login`) или при предыдущем вызове refresh. |

Пример тела:

```json
{
  "refreshToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
```

**Успешный ответ** — `200 OK`

Тело — та же структура, что и у `POST /v1/auth/login`: поля `accessToken`, `refreshToken` (новый, при ротации), `expiresIn`, опционально `user`. Клиент сохраняет новые токены и повторяет исходный запрос.

**Ошибки**

| Код | Описание |
|-----|----------|
| 401 | Refresh token отсутствует, истёк или невалиден. Клиент очищает токены и перенаправляет на посадочную (вход через Telegram). |
| 500 | Внутренняя ошибка сервера. |

## Поведение клиента (кратко)

1. **При старте (Mini App):** отправить `initData` на `POST /v1/auth/login`, сохранить `accessToken` и `refreshToken`.
2. **При входе (браузер):** после авторизации в Telegram отправить данные на `POST /v1/auth/login`, сохранить токены.
3. **При запросе к защищённому эндпоинту:** заголовок `Authorization: Bearer <accessToken>`.
4. **При ответе 401:** вызвать `POST /v1/auth/refresh` с сохранённым `refreshToken`; при 200 — сохранить новые токены и повторить запрос; при 401 — очистить токены и редирект на посадочную.
5. **При выходе (кнопка «Выйти»):** вызвать `POST /v1/auth/logout` с текущим `refreshToken`, очистить токены, перенаправить на посадочную (в браузере кнопка «Выйти» — в Настройках).
