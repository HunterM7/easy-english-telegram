# API: Авторизация

[Docs](../../README.md) → [Server](../README.md) → [API](README.md) → **Авторизация**

## Текущий этап: только Telegram Mini App

На **первом этапе** продукт рассчитан на работу **только внутри Telegram Mini App**. Вход — **авторизация через Telegram** по строке **`initData`** от Web App: бэкенд проверяет подпись и выдаёт **`accessToken`** (JWT) и **`refreshToken`** (непрозрачный токен, см. [модель токенов](#модель-токенов-access-jwt-и-refresh-opaque)). Дальнейшие запросы к API выполняются с **`accessToken`** в заголовке; при его истечении клиент обновляет сессию по **`refreshToken`**, не требуя повторного входа.

**Открытие в обычном браузере** (по прямой ссылке на фронт) **не предполагает** входа в приложение и работы с API: пользователь видит [посадочную страницу](../../client/screens/landing.md) с переходом к **боту**, через которого запускается Mini App. Полноценная авторизация в браузере (чтобы пользоваться тем же приложением без Telegram-клиента) — **в планах**, см. раздел [«Планы»](#планы-браузерная-сессия-и-выход) ниже.

## Модель токенов: `accessToken` (JWT) и `refreshToken` (opaque)

Ниже — зафиксированное **продуктово-техническое решение**: почему два разных типа токенов, как они устроены и как ведут себя на сервере. Это нужно, чтобы не путать «два JWT» и понимать, зачем в БД хранится `refreshToken`.

### Термины

| Термин | Что это |
|--------|---------|
| **`accessToken`** | Короткоживущий **JWT** (JSON Web Token). Подписан секретом сервера (`JWT_SECRET`). Содержит минимум данных (например, идентификатор пользователя в `sub`). Используется **только** в заголовке `Authorization: Bearer …` для доступа к защищённым эндпоинтам. |
| **`refreshToken`** | Долгоживущий **непрозрачный (opaque) токен** — случайная строка (в реализации — например UUID), **не** JWT. Сервер при выдаче **сохраняет** её в базе данных, связывая с пользователем и сроком действия. Клиент передаёт её **только** в теле `POST /v1/auth/login` (косвенно — при первом входе её ещё нет) и `POST /v1/auth/refresh`. В JSON ответа поле **`refreshToken`**; **формат строки — произвольный идентификатор сессии**, а не JWT. |

### Почему `accessToken` — JWT, а `refreshToken` — не JWT

**`accessToken` (JWT)** удобен для **каждого HTTP-запроса** к API:

- Сервер проверяет **подпись** и срок действия **без обращения к БД** (stateless), что снижает нагрузку при большом числе запросов.
- Срок жизни короткий (рекомендуется ~30 минут) — при утечке окно риска ограничено.

**`refreshToken`** решает другую задачу: **долгая сессия** без повторного входа через Telegram. Здесь важны:

1. **Отзыв и контроль**  
   Если сессию нужно прекратить (истёк срок, подозрение на утечку, смена устройства в будущем), сервер должен иметь **авторитетный источник правды**. Для opaque-токена достаточно **удалить или пометить запись в БД** — переданный **`refreshToken`** перестаёт работать сразу для всех инстансов приложения.

2. **Ротация `refreshToken`**  
   При каждом успешном `POST /v1/auth/refresh` старый **`refreshToken`** инвалидируется (удаляется), выдаётся **новый** opaque-токен и новая запись в БД. Укравший старый **`refreshToken`** не может бесконечно продлевать сессию после одного успешного обновления.

3. **Путаница с «двумя JWT»**  
   Если сделать **и `accessToken`, и `refreshToken`** JWT без хранения **`refreshToken`** в БД, проверка **`refreshToken`** сводится к проверке подписи и срока — но **отозвать конкретную сессию** до истечения срока сложнее (нужен blacklist JWT, отдельное хранилище или очень короткий TTL). Тогда снова появляется потребность в **хранилище** на сервере. Проще изначально хранить **`refreshToken`** как **идентификатор строки в БД**.

**Итог:** для нашего стека (NestJS + PostgreSQL + Prisma) выбрана схема **JWT только для `accessToken`** и **opaque `refreshToken` в БД** — баланс простоты, безопасности и возможности отзыва сессии без изобретения отдельной инфраструктуры blacklist для JWT.

### Как это работает по шагам (сервер)

1. **Успешный `POST /v1/auth/login` с валидным `initData`**  
   - Создаётся/обновляется пользователь.  
   - Генерируется JWT для **`accessToken`** с коротким TTL.  
   - Генерируется случайное значение **`refreshToken`**, сохраняется в таблице (например `refresh_tokens`: токен, `user_id`, `expires_at`).  
   - Клиенту возвращаются оба значения в JSON.

2. **Запрос к защищённому API**  
   - Клиент посылает только JWT **`accessToken`** в `Authorization`.  
   - Guard проверяет подпись и срок JWT; при необходимости загружает пользователя по `sub`.

3. **Истёк `accessToken`, клиент вызывает `POST /v1/auth/refresh`**  
   - По телу запроса приходит строка **`refreshToken`**.  
   - Сервер ищет запись в БД: токен существует, не истёк, привязан к пользователю.  
   - Запись старого **`refreshToken`** удаляется (ротация).  
   - Выдаются **новый JWT `accessToken`** и **новое значение `refreshToken`** с новой записью в БД.

4. **Невалидный / истёкший `refreshToken`**  
   - Ответ **401**. Клиент очищает локальные токены; в Mini App пользователю показывается сценарий повторного входа через Telegram (новый `initData`).

### Замечание про «шифрование» JWT

Подпись JWT **не шифрует** содержимое для клиента: полезная нагрузка обычно кодируется в Base64 и может быть прочитана при желании. Безопасность опирается на **подпись** (нельзя подделать без `JWT_SECRET`) и на **HTTPS** при передаче. В payload **`accessToken`** не кладут секреты; **opaque-значение `refreshToken` не должно нести критичной логики** — его сила в том, что **валидно только если есть запись в БД**.

### Связь с полями API

В JSON ответов контракт остаётся прежним: `accessToken`, `refreshToken`. Семантика:

- `accessToken` — всегда **JWT** (три части через точку, как в примерах ниже по смыслу **`accessToken`**).  
- `refreshToken` — **строка-секрет**, формат на усмотрение бэкенда (UUID и т.д.), **не** обязательно похожа на JWT.

## Схема работы (Mini App)

### Вход (получение токенов)

**Mini App:** при старте приложения клиент получает от Telegram Web App строку `initData`, отправляет её на бэкенд в теле `POST /v1/auth/login`. Сервер проверяет подпись (валидность данных от Telegram), создаёт или находит пользователя в БД и возвращает `accessToken` и `refreshToken`.

Клиент сохраняет оба токена (в памяти или безопасном хранилище, допустимом в WebView) и при каждом запросе к защищённым эндпоинтам передаёт значение **`accessToken`** в заголовке `Authorization: Bearer <accessToken>`.

### Сроки жизни токенов

| Токен            | Рекомендуемый срок | Назначение |
|------------------|---------------------|------------|
| **`accessToken`** | 30 минут            | Подпись запросов к API; короткий срок снижает риски при утечке. |
| **`refreshToken`** | до 30 дней (запись в БД) | Обновление **`accessToken`** без повторного входа; фактическое значение — opaque-токен на сервере (см. [модель токенов](#модель-токенов-access-jwt-и-refresh-opaque)); при истечении или отзыве — снова Mini App с новым `initData`. |

Конкретные значения задаются на бэкенде (переменные окружения или конфиг).

### Обновление сессии (refresh)

Когда защищённый эндпоинт возвращает **401 Unauthorized** (**`accessToken`** истёк или невалиден), клиент **не** перенаправляет пользователя на вход сразу. Он:

1. Отправляет запрос **`POST /v1/auth/refresh`** с телом `{ "refreshToken": "..." }` (значение сохранённого **`refreshToken`**).
2. Если ответ **200 OK** — сохраняет новые `accessToken` и при необходимости `refreshToken` из ответа, затем **повторяет исходный запрос** к API с новым **`accessToken`**.
3. Если ответ **401** (**`refreshToken`** истёк или невалиден) — очищает сохранённые токены и показывает пользователю **экран с объяснением** (например, сессия устарела) и **ссылку/кнопку на бота**, чтобы снова открыть Mini App. Отдельной «посадочной с авторизацией» на первом этапе нет.

### Публичные и защищённые эндпоинты

- **Публичные** (без **`accessToken`** в заголовке): `POST /v1/auth/login`, `POST /v1/auth/refresh`.
- **Защищённые**: все остальные эндпоинты API требуют заголовка `Authorization: Bearer <accessToken>`.

## Планы: браузерная сессия и выход

**Не входит в текущее ТЗ.** Ниже — ориентир для последующей версии (вместе с [посадочной и входом в браузере](../../client/screens/landing.md)).

1. **Вход в браузере:** если пользователь открывает **веб-приложение в браузере** (не внутри Telegram), дать возможность **войти через Telegram** (например, [Telegram Login Widget](https://core.telegram.org/widgets/login), OAuth/редирект в Telegram) и получить **`accessToken`** и **`refreshToken`** по той же [модели](#модель-токенов-access-jwt-и-refresh-opaque), что и в Mini App. Посадочная сможет вести не только к боту, но и к сценарию «войти → тот же интерфейс, что в Mini App». Контракт `POST /v1/auth/login` может быть расширен отдельным телом запроса для виджета/браузера — уточнить при реализации.

2. **Выход из сессии в браузере:** кнопка **«Выйти»** в настройках; очистка **`accessToken`** и **`refreshToken`** на клиенте; редирект на посадочную при невалидном **`refreshToken`** после истечения сессии. На бэкенде — инвалидация **`refreshToken`** (например, эндпоинт `POST /v1/auth/logout` с телом `{ "refreshToken": "..." }`, ответ 200/204). Конкретный контракт и необходимость чёрного списка токенов — уточнить при реализации; **в текущем этапе эндпоинт и UI не проектируем**.

## Эндпоинты

### POST /v1/auth/login

Вход в приложение. Обмен данных верификации Telegram на **`accessToken`** (JWT) и **`refreshToken`** (opaque, см. [модель токенов](#модель-токенов-access-jwt-и-refresh-opaque)).

**Текущий этап:** вызывается при старте Mini App после получения `initData` от Telegram Web App.

**Запрос**

| Поле       | Тип    | Обязательно | Описание |
|------------|--------|-------------|----------|
| `initData` | string | да          | Строка initData от Telegram Web App. Содержит данные пользователя и подпись; бэкенд проверяет подпись с помощью секрета бота (см. документацию Telegram). |

Пример тела:

```json
{
  "initData": "query_id=AAHdF6IQAAAAAN0XohAOq..."
}
```

**Успешный ответ** — `200 OK`

| Поле           | Тип    | Описание |
|----------------|--------|----------|
| `accessToken`  | string | **JWT** для заголовка `Authorization: Bearer <token>`. Срок жизни — 30 мин (рекомендуется). |
| `refreshToken` | string | **Непрозрачный** токен (не JWT), хранится на сервере; для обновления **`accessToken`** через `POST /v1/auth/refresh`. Срок жизни сессии — до 30 дней (рекомендуется), задаётся записью в БД. |
| `expiresIn`    | number | (опционально) Время жизни **`accessToken`** в секундах. |
| `user`        | object | (опционально) Данные пользователя для отображения на клиенте. |

Пример ответа:

```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "refreshToken": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "expiresIn": 1800,
  "user": {
    "id": "telegram-user-id",
    "firstName": "Имя",
    "lastName": "Фамилия",
    "username": "username",
    "photoUrl": "https://..."
  }
}
```

**Ошибки**

| Код | Описание |
|-----|----------|
| 400 | Неверный или поддельный initData (проверка подписи не прошла). |
| 401 | Не удалось верифицировать пользователя Telegram. |
| 500 | Внутренняя ошибка сервера. |

Тело ошибки (единый формат API):

```json
{
  "statusCode": 400,
  "message": "Invalid Telegram initData",
  "error": "Bad Request"
}
```

### POST /v1/auth/refresh

Обновление **`accessToken`** по действительному **`refreshToken`**. Вызывается клиентом при получении 401 от любого защищённого эндпоинта: сначала попытка обновления; при успехе — повтор исходного запроса; при 401 от **`refreshToken`** — очистка токенов. **В Mini App** далее — экран с призывом открыть приложение из Telegram через бота, а не редирект на посадочную как на экран входа. **В будущей браузерной версии** — редирект на посадочную (вход через Telegram).

**Запрос**

| Поле           | Тип    | Обязательно | Описание |
|----------------|--------|-------------|----------|
| `refreshToken` | string | да         | Значение, полученное при входе (`POST /v1/auth/login`) или при предыдущем вызове `POST /v1/auth/refresh`. |

Пример тела:

```json
{
  "refreshToken": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

**Успешный ответ** — `200 OK`

Тело — та же структура, что и у `POST /v1/auth/login`: поля `accessToken` (новый JWT), `refreshToken` (новый opaque-токен после **ротации** — старый **`refreshToken`** после успешного вызова недействителен), `expiresIn`, опционально `user`. Клиент сохраняет новые значения и повторяет исходный запрос.

**Ошибки**

| Код | Описание |
|-----|----------|
| 401 | Поле **`refreshToken`**: отсутствует, истёк или невалиден. Клиент очищает токены; в Mini App — UI с переходом к боту; в браузерной версии (когда будет) — посадочная. |
| 500 | Внутренняя ошибка сервера. |

## Поведение клиента (кратко)

1. **При старте (Mini App):** отправить `initData` на `POST /v1/auth/login`, сохранить значения полей `accessToken` и `refreshToken`.
2. **При запросе к защищённому эндпоинту:** заголовок `Authorization: Bearer <accessToken>`.
3. **При ответе 401:** вызвать `POST /v1/auth/refresh` с телом `{ "refreshToken": "…" }` (последнее сохранённое значение); при 200 — сохранить новые `accessToken` и `refreshToken` и повторить запрос; при 401 — очистить **`accessToken`** и **`refreshToken`** и показать экран с переходом к боту (Mini App).
4. **Браузер без Mini App:** до внедрения отдельной авторизации — только посадочная и ссылка на бота; **после** — см. [планы](#планы-браузерная-сессия-и-выход).

См. также [PRD, раздел 4.5](../../product/PRD.md) и [implementation: авторизация](../implementation.md).
